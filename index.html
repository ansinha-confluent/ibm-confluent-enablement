Overview

Confluent × IBM: the technical integration, end to end

IBM closed its acquisition of Confluent on 17 March 2026 (~$11B, $31/share). This guide is the SE-to-SE version: where the two stacks actually bolt together, the configs that matter, and the places the seams still show. No pitch — just the wiring.

If you already sell or deploy Kafka and Flink, the mental shift is small. Confluent stays what it is — the real-time substrate. What changes is everything downstream and around it: a governed lakehouse (watsonx.data), an agent/reasoning layer (watsonx.ai + Orchestrate), governance and lineage spanning the whole thing, and first-class paths off the mainframe. IBM's own framing for the deal is deliberately additive: "connect, process and govern real-time data for applications and AI agents."

Independent read

Forrester's take (Mar 2026): IBM paid for a real-time data substrate for AI agents, not for Kafka as a product line. The structural tell — unlike the Red Hat acquisition, Confluent is not being run as an arm's-length independent unit. It's being woven into watsonx, hybrid integration, and Z. For you that means the integration points below are the strategy, not afterthoughts.

The three-layer model

Almost every IBM+Confluent architecture you'll draw collapses to three layers plus a governance spine. Internalize this and the four pillar pages stop feeling like separate products.

  1. Nervous system — Confluent. Kafka is the durable event log; Flink does the sub-second processing; Connectors + Tableflow get data in and out. This is where "now" lives.
  2. Data foundation — IBM watsonx.data. An open lakehouse: Iceberg + Parquet tables, queried by Presto (C++/Velox) and Spark, with Milvus alongside for vectors. This is where "everything, organized and governed" lives.
  3. Reason & act — watsonx.ai + watsonx Orchestrate. Granite models and RAG for reasoning; Orchestrate for multi-step agentic workflows with human-in-the-loop and MCP tool calls.

Spanning all three: watsonx.governance and watsonx.data intelligence (lineage). Governance that only covers the lake but not the streams feeding it is theater — the point of the spine is that it reaches into the event layer too.

When to use what

The single most useful thing you can give a customer is a clean decision rule for which layer owns a workload. Here's the one that holds up:

If the workload is…It belongs to…Why
Sub-minute real-time context, stream processing, event distributionConfluent / FlinkLatency and replay are first-class; Flink does stateful processing in motion
Multi-step business workflow, human-in-the-loop, tool/agent orchestrationwatsonx OrchestrateDurable, supervised, long-running — not a stream job
Historical / at-rest analytics, federated SQL, AI training datawatsonx.data lakehouseOpen tables, cost-efficient engines, governed once

Most real designs use all three and the interesting work is the handoffs — which is exactly what the watsonx.data page is about (Tableflow is the handoff from layer 1 to layer 2).

Shared open standards = less lock-in

The integration leans on formats both companies already standardized on, which is what makes the handoffs cheap: Apache Kafka, Apache Flink, Apache Iceberg, Apache Parquet, Presto, and MCP (Model Context Protocol) for tool/agent wiring. When a customer worries about lock-in, this is the honest answer: the table format and the streaming engine are open; the value-add is the managed surfaces around them.

What's GA vs. preview (be precise with customers)

  • GA: Confluent + Tableflow + Flink integrations with watsonx.data (announced at Think, 5 May 2026); the IBM MQ connectors; Confluent Platform certified on Linux on Z / s390x; IBM Confluent Platform as a product (standalone and as a CP4I / webMethods add-on).
  • Preview / emerging: "Context in watsonx.data" federated semantic layer; next-gen multi-agent watsonx Orchestrate control plane; GPU-accelerated Presto; native watsonx.data Sink connector and the Real-Time Context Engine. Label these as forward-looking when you quote them.
Accuracy note carried into the Integration page

There is no public formal Statement of Direction deprecating IBM Event Streams or Event Processing. IBM still actively ships and updates them. The correct framing is "IBM Confluent Platform is the strategic go-forward streaming engine and the natural consolidation target" — not "Event Streams is EOL." See the IBM Integration page for the full picture.

Each opens with an architecture diagram, goes deep on components/config/gotchas, then works 2–3 named use cases with their own diagrams, and ends with cited references.

Pillar 1

Confluent + IBM watsonx.data

The short version: watsonx.data is an open lakehouse — Iceberg + Parquet tables queried by Presto (C++/Velox) and Spark, running on IBM Cloud / AWS / Azure / GCP and on-prem (OpenShift, IBM Storage Fusion). The integration spine is Confluent Tableflow: it continuously materializes a Kafka topic into Iceberg, zero-copy, so you don't stand up and babysit a second ETL pipeline to land streaming data in the lake. The interesting parts are the catalog wiring, a hard read-only constraint, and where managed-storage convenience stops.

How Tableflow actually works

Tableflow watches a Kafka topic and keeps an Iceberg (or Delta) table in sync with it. It reads the topic's schema from Schema Registry, converts Avro / JSON Schema / Protobuf records to Parquet, and handles automated schema evolution (add/remove columns, type widening), CDC materialization, and Parquet compaction so you're not left with a swamp of tiny files. You get a built-in Iceberg REST Catalog, or you sync the tables out to an external catalog — AWS Glue, Snowflake Open Catalog (Polaris), or Databricks Unity. Storage is either Confluent managed storage or BYOS (bring-your-own — S3 or IBM COS).

Tableflow scope limits

Tableflow is Confluent Cloud only today — not Confluent Platform, and not GCP-backed clusters at GA. Snapshot retention is configurable only within 10–100 snapshots. If a customer's mental model is "Tableflow everywhere including my on-prem CP cluster," reset it early. (On-prem materialization is emerging via WarpStream Tableflow + watsonx.data on-prem — flag it as forward-looking.)

Wiring watsonx.data to Tableflow — the exact config

watsonx.data registers the Tableflow Iceberg REST catalog as a data source. Two engines, two configs, two sets of footguns.

Spark engine

spark.sql.catalog.tableflow            = org.apache.iceberg.spark.SparkCatalog
spark.sql.catalog.tableflow.type       = rest
spark.sql.catalog.tableflow.uri        = https://tableflow.{REGION}.aws.confluent.cloud/iceberg/catalog/organizations/{ORG_ID}/environments/{ENV_ID}
spark.sql.catalog.tableflow.credential = {APIKEY}:{SECRET}
spark.sql.catalog.tableflow.io-impl    = org.apache.iceberg.aws.s3.S3FileIO
spark.sql.catalog.tableflow.s3.remote-signing-enabled = true   # required for managed storage
# namespace = your Kafka cluster id (lkc-xxxxx); table name = the topic name

The catalog uri is the Tableflow REST endpoint scoped to your org and environment; the credential is a Cloud API key/secret pair. With managed storage, s3.remote-signing-enabled=true is what lets Spark read the data files without you handing it raw bucket credentials — Confluent signs the requests.

Presto engine

connector.name         = iceberg
iceberg.catalog.type   = REST
iceberg.rest.uri       = <tableflow catalog uri>
iceberg.rest.auth.type = OAUTH2
Presto gotchas (the ones that bite)

1. Presto does not support Confluent managed storage / vended credentials — so with Presto you must use BYOS. 2. Presto can't auto-discover Tableflow tables; you have to reference exact topic/table names. 3. The namespace is the cluster id (lkc-xxxxx) which contains hyphens — quote it in SQL. Provision it as a custom Iceberg data source in watsonx.data Infrastructure Manager.

The read-only constraint — design around it

Verbatim from Confluent docs

"Iceberg tables exposed via external catalog synchronization are read-only. When working with Iceberg tables created by Tableflow, it's essential to ensure that they are consumed as read-only tables."

So: no INSERT / UPDATE / DELETE against these tables from Spark or Presto. The write path is the stream — you change the data by publishing to the source Kafka topic, and Tableflow reflects it. This is the right model (single source of truth is the log) but it surprises people who treat the lakehouse table as mutable. Note also that external catalog sync (Glue / Polaris / Unity) requires BYOS — it doesn't work with managed storage.

Land it clean: medallion + Flink AI functions

Don't dump raw bronze into the lake and clean it later. Use Flink to filter, dedupe, and enrich before Tableflow, so data lands silver/gold. Confluent Cloud Flink ships a useful set of in-SQL AI/ML functions you can run inline in that shaping step:

  • Classical ML: ML_PREDICT, ML_FORECAST, ML_DETECT_ANOMALIES (ARIMA), ML_DETECT_ANOMALIES_ROBUST (MAD).
  • Generative / foundation-model: AI_COMPLETE, AI_EMBEDDING, AI_SENTIMENT, AI_DETECT_PII, AI_DETECT_ANOMALIES (TimesFM), AI_RUN_AGENT, AI_TOOL_INVOKE.

Vectors, lineage, reasoning

  • Milvus in watsonx.data — managed vector store (the watsonx-data reference cites up to ~100M vectors × 384 dims). Run it as a parallel sink alongside Iceberg: unstructured text → embeddings → Milvus for semantic search / RAG, wired into watsonx Orchestrate.
  • watsonx.data intelligence (= IBM Data Lineage, integrating Manta) does OpenLineage-based harvesting across 50+ integrations, column-level end-to-end lineage, AI factsheets, quality and governance.
  • watsonx.ai Granite models do the reasoning — e.g. pulling structured codes out of free-text notes (see the healthcare use case).
  • MCP: watsonx.data 2.2.x ships a remote MCP server; import it into Orchestrate with orchestrate toolkits import --kind mcp.
Honest hedge

IBM docs name OpenLineage and "50+ integrations," but I couldn't verbatim-confirm a documented "harvests Confluent Schema Registry" path. Phrase it to customers as "harvests operational metadata from streaming sources, including schemas" rather than claiming a named Schema-Registry lineage connector.

Use cases

★ UC-A — Real-time insurance claims fraud (catastrophe surge) flagship

After a hurricane, claims spike and fraud peaks exactly when adjusters are buried. Nightly batch scoring is useless here — you need to flag suspect claims as they arrive. IBM MQ carries the claims; the Confluent IBM MQ Source connector lands them on a Kafka claims topic; Flink scores them inline with ML_DETECT_ANOMALIES / AI_DETECT_ANOMALIES; the enriched stream flows through Tableflow to Iceberg; watsonx.data Presto answers the at-rest questions (approvals/denials by city, 6-month fraud look-back); and watsonx Orchestrate agents work the flagged queue.

Components: IBM MQ Source connector, Confluent Cloud Flink (anomaly functions), Tableflow (BYOS on S3/COS), watsonx.data Presto, watsonx Orchestrate. Honesty: the Flink functions are real and cited; the specific demo writeup isn't public, so present this as a realistic reference architecture, not a shipped IBM demo.

★ UC-B — Customer 360 with live sentiment (federated join) flagship

Clickstream and session events stream into Kafka; Flink sessionizes them and scores sentiment with AI_SENTIMENT; Tableflow lands a live session table in Iceberg. Then watsonx.data Presto does the thing lakehouses are good at — a federated join of the fresh session table against historical demographics that already live in the lake:

SELECT s.session_id, s.sentiment_score, d.segment, d.lifetime_value
FROM   tableflow."lkc-xxxxx".sessions_live  s
JOIN   hive.demographics.customer_360         d
  ON   s.customer_id = d.customer_id
WHERE  s.sentiment_score < 0.3;   -- unhappy, high-value sessions, right now

The win is no second copy and no reverse-ETL: the streaming table and the historical table sit in the same query engine. Mind the cluster-id namespace quoting (the gotcha above).

UC-C — Healthcare clinical-code extraction (governance vehicle)

Clinical events stream to Kafka → Tableflow → Iceberg → watsonx.data, where a Granite model on watsonx.ai extracts ICD-10 codes from free-text notes and reconciles them against billing. The point of this one is governance: it exercises end-to-end column-level lineage via watsonx.data intelligence and makes the read-only projection constraint concrete (the lake table is the governed read model; corrections go back through the stream).

Design considerations & gotchas (quick reference)

Before you commit a design

• Tableflow = Cloud-only, not CP, not GCP clusters (GA).
• Presto ⇒ BYOS mandatory; no managed-storage vended creds; no table auto-discovery; quote the lkc- namespace.
• Tables are read-only; the write path is the topic.
• External catalog sync (Glue/Polaris/Unity) needs BYOS.
• Shape data with Flink to land silver/gold; snapshot retention is 10–100.
• Schema-Registry-as-lineage-source is a hedge, not a documented connector — phrase accordingly.

Pillar 2

Confluent + IBM Integration

The short version: IBM now ships IBM Confluent Platform as a product — standalone and as an add-on to Cloud Pak for Integration (CP4I) and webMethods. Around it sits IBM's existing hybrid-integration estate: Event Automation (Event Streams / Event Processing / Event Endpoint Management), IBM MQ, App Connect, and IBM Data Integration (StreamSets, DataStage, Data Replication, Databand). This page is about how Confluent slots into that estate, the MQ connector trade-offs you'll actually argue about, and one correction worth getting right.

The correction (say this accurately)

There is no public formal Statement of Direction deprecating Event Streams or Event Processing. As of mid-2026 IBM still markets and actively develops them (Event Streams shipped Kafka 3.8 and CLI updates in Apr 2026). What is deprecated: "Event Streams for IBM Cloud Satellite," and the CP4I-7.0.x Confluent packaging hit EOL in Apr 2025. Correct framing: IBM Confluent Platform is the strategic go-forward streaming engine and the natural consolidation target; EEM / AsyncAPI is the governance layer that carries forward. Do not quote a deprecation date — there isn't one, and SEs who invent one lose credibility.

The Event Automation layer (on top of Confluent Kafka)

  • Event Streams — IBM's Kafka distribution. The consolidation story moves this role toward IBM Confluent Platform over time, but it's alive today.
  • Event Processing — a drag-and-drop, Flink-style low-code authoring UI. This is the genuine differentiator versus raw Confluent: a no-code surface for building stream processing that Confluent doesn't ship natively.
  • Event Endpoint Management (EEM) — an AsyncAPI catalog plus a socialization/governance gateway. This carries forward regardless of which Kafka engine sits underneath.

Public scale references: PVcomBank ~5M notifications/day; Ajman ~50K TPS; Norsk Tipping 5K+ gaming TPS.

IBM MQ ↔ Confluent: coexistence and honest connector trade-offs

This is the most common conversation. MQ is mission-critical transactional messaging; Confluent adds replay, fan-out to many independent consumers, stream processing, and event distribution at scale. They coexist — the decision is use-case-led, not religious. The connectors are where the detail lives:

ConnectorVersion / packagingNotes
IBM OSS MQ Source / Sinksource v2.8.1, sink v2.1.2 (Apache-2.0)CLIENT or BINDINGS mode; exactly-once on Kafka 3.3+ via a dedicated state queue (tasks.max=1); XML record builder; MQMD access
Confluent Cloud fully-managed IBM MQ Sinkmanagedport 1414 default; CSFLE; RFH2 suppression (mq.destination.suppress.rfh2); exactly-once via dedicated offsets queue (EXCLUSIVE, single task); base64 TLS keystore
Confluent Platform self-managed MQ Source/Sinkself-managed; s390x from CP 8.2+HA via mq.connection.list; BINDINGS via JNI

A minimal self-managed IBM MQ Source connector looks like this:

"connector.class": "io.confluent.connect.ibm.mq.IbmMQSourceConnector",
"kafka.topic":     "claims.in",
"mq.queue.manager": "QM1",
"mq.connection.mode": "BINDINGS",        // or CLIENT (TCP/IP)
"mq.queue":         "CLAIMS.REQUEST",
"tasks.max":        "1"                  // required for exactly-once
Opinionated trade-off

Lead with Confluent for cloud/elastic deployments, fully-managed connectors, and Schema Registry + CSFLE. Lead with the IBM/native connectors for z/OS bindings mode, deep MQ-specific configuration, and when the customer wants IBM-supported Kubernetes operator artifacts. On z/OS specifically, use the certified IBM MQ connectors in bindings mode — that's covered on the IBM Z page.

webMethods, App Connect, and IBM Data Integration vs. Confluent DSP

  • webMethods Hybrid Integration — IBM's iPaaS, with an Event Automation component; the SaaS complement to CP4I.
  • App Connect Enterprise (ACE) 13.x — 200+ connectors; can expose flows and connectors as MCP tools, which matters for the agent layer.
  • IBM Data Integration — StreamSets (JVM visual low-code pipelines: origins → processors → destinations, with data-drift auto-adapt; embedded in watsonx.data integration), DataStage (batch ETL/ELT), Data Replication / IIDR (CDC from Db2 z/OS, VSAM, IMS → Kafka), Databand (observability).
Where the line falls

Lead with IBM Data Integration for batch ETL, heterogeneous database movement, and data-drift-tolerant pipelines. Lead with Confluent DSP (Kafka commit log + Flink + Tableflow + Connect + governance) for sub-minute event streaming, replay, and event-native processing. They overlap in the middle; the deciding question is "is the unit of work an event or a batch?"

Connectors & distribution

  • Db2 Source (JDBC) for LUW — fully-managed GA on Confluent Cloud; snapshot + incrementing/timestamp modes; Avro/JSON_SR/Protobuf/JSON; CSFLE.
  • Oracle CDC (XStream) — premium connector, cloud and platform.
  • SAP S/4HANA — two phases: Phase 1 OData v4 polling (fully-managed); Phase 2 push via ABAP partner add-ons → HTTP/Webhook Source. ASAPIO is confirmed (REST Proxy + Webhook, Confluent-certified since Apr 2021); SNP/Onibex are named but not separately verified.
  • Distribution — Confluent on-prem via IBM Passport Advantage Online. Confluent's public Docker image reference lists amd64 + arm64; s390x builds are delivered through IBM distribution (CP is broadly certified on Linux on Z — see the IBM Z page).

Use cases

★ UC-A — Event Streams / CP4I → IBM Confluent Platform consolidation flagship

Before: CP4I-embedded Event Streams (Kafka) + Event Processing + EEM. After: IBM Confluent Platform as the streaming engine, with EEM/AsyncAPI carried forward as the governance layer and Event Processing's low-code surface on top. Frame this as the steered consolidation path — the direction IBM is pointing customers — not a forced EOL migration with a deadline.

★ UC-B — webMethods → Confluent Cloud → watsonx.data (data availability) flagship

webMethods publishes to a Confluent Cloud topic; an optional light Flink step shapes it; Tableflow lands it in Iceberg; it shows up in watsonx.data. The goal here is data-availability latency — getting webMethods-integrated data into the lakehouse fast — not query benchmarking. It's the cleanest way to make iPaaS-sourced data analytics-ready without a bespoke pipeline.

UC-C — MQ modernization / event distribution at scale

Keep the MQ backbone for transactional messaging, bridge it to Confluent with the MQ Source/Sink connectors, and let downstream microservices, analytics (watsonx.data), and AI agents all consume the same governed streams — with replay and fan-out that MQ alone doesn't give you. Be honest about the connector trade-offs (table above): bindings mode and MQ-specific config favor the native connectors; managed + Schema Registry favors Confluent's.

Pillar 3

Confluent + IBM Z

The short version: IBM Z is the z/Architecture mainframe; its Linux architecture is s390x (64-bit), running z/OS, Linux on Z, or LinuxONE. The cost lever everyone cares about is the zIIP (Z Integrated Information Processor) — a specialty engine for eligible Java workloads whose capacity doesn't count toward MIPS and isn't chargeable under workload-based software maintenance. Push Confluent's Java connectors onto zIIP and you move real-time mainframe integration off the general-purpose engines you're billed on. ~70%+ of the Fortune 500 (and ~92% of the top-100 banks) still run mainframes, so this is where a lot of the money is.

Connect on z/OS — the verified specifics

Confluent certifies Kafka Connect to run on z/OS, hosting the certified premium IBM MQ Source v12.1.1 and Sink v2.1.2. The detail that matters for cost and security is the connection mode:

Run it in BINDINGS mode

On z/OS, run the MQ connectors in BINDINGS mode (JNI, in-memory, local to the MQ address space — MQCNO_STANDARD_BINDINGS). This avoids routing through the CHINIT address spaces that CLIENT (TCP/IP) mode requires → more secure, and far less networking + CHINIT consumption. CLIENT mode is the fallback when Connect runs off-platform.

z/OS specifics you'll hit during setup: EBCDIC IBM-1047 via iconv; runs under USS + Bash; the Java workload is zIIP-eligible; Log4j2 charset set to IBM-1047; JCL redirects STDOUT/STDERR to USS files; certified on z/OS v2.4; needs com.ibm.mq.allclient.jar. Use v12.x (at-least-once); v11.x is deprecated.

Value & outcomes — label internal vs. public

Be disciplined about which numbers are published

Public (Confluent blog / MIPS whitepaper): zIIP offload reduces processing cost; the pattern vastly cuts MQ networking + CHINIT cost; named customers Alight, RBC, KeyBank. Internal pilot figures (present as such, not as published doc claims): ~10× cheaper / ~10× faster; ~30K msg/s at ~3KB with ~$1.2M projected savings; another engagement cut MQ MIPS by ~90%. If you put the internal numbers on a slide, label them "internal pilot," and cite the public blog for the directional claim.

Mainframe sources & CDC options

Sources: Db2 for z/OS, VSAM, IMS, CICS/COBOL, IBM MQ. The CDC choice is the real architecture decision:

  • IIDR (IBM Data Replication) — reads Db2 active logs; the canonical IBM path to stream z/OS Db2 → Kafka; also handles VSAM/IMS. Cases: Garanti BBVA, a Middle East bank VSAM CIF. Carries licensing cost.
  • IBM Data Gate for Confluent — log-based CDC from Db2 z/OS to Confluent, emitting Debezium-compatible events, 96% zIIP-eligible, with initial load + continuous CDC, Avro/JSON + Schema Registry + TLS. Publicly announced 2026-05-05 — present as "announced May 2026, GA mid-2026" (don't over-pin the GA date).
  • zDIH (IBM Z Digital Integration Hub) — an in-memory cache of z/OS systems-of-record that also publishes events to Kafka; shields production from inquiry traffic; zIIP-optimized. Case: M&T Bank.
  • CICS/COBOL pattern — CICS → CICS-MQ adapter → IBM MQ → self-managed IBM MQ Source connector on z/OS → Kafka.
  • Third-party CDC (mention only, where IBM tooling isn't licensed): Precisely, Oracle GoldenGate, Informatica, Qlik, tcVision.
The honest gap

There is no fully-managed Confluent connector for Db2 z/OS yet. Today the real options are IIDR (licensed) or IBM Data Gate (newly announced). Don't imply a one-click managed Db2-z/OS source exists.

Confluent Platform on s390x / LinuxONE + hybrid

IBM support states "IBM Confluent Platform for Z and LinuxONE is broadly certified to run on Linux on Z (s390x)," covering the majority of features, with multi-arch Docker images on OpenShift/Kubernetes via CFK (Confluent for Kubernetes). Deploy on LinuxONE, Linux on Z, or zCX (which runs on an IFL or zIIP).

Hedge on the component list

The full per-component certification list (ksqlDB / Control Center / REST Proxy / Flink) isn't individually enumerated publicly — say "core CP components," not a specific checklist, unless you've confirmed it for the customer's version.

Hybrid topology: run core streaming on Z/LinuxONE and use Cluster Linking to mirror topics to Confluent Cloud — byte-for-byte, offset-preserving, on-prem CP 7.0+ → Cloud. That keeps low-latency mainframe-adjacent processing on-platform while the cloud side feeds watsonx.data and elastic consumers. For consolidation economics, the Signal65/IBM LinuxONE TCO work is the citable source (e.g., 47 x86 cores → 11 IFLs).

Don't overstate the case studies

The Fiserv LinuxONE-specific Confluent case is not citable — Fiserv's Signature modernization is on Power10, not LinuxONE. Use Signal65 for LinuxONE consolidation numbers and the Confluent zSystems blog (Alight/RBC/KeyBank) for the streaming story.

Use cases

★ UC-A — Mainframe MQ offload for cost + real-time access flagship

A CICS application puts messages on MQ on z/OS; the certified IBM MQ Source connector — running on zIIP in bindings mode — lands them on Kafka off the mainframe, where cloud apps and analytics consume them. Outcomes: the public zIIP cost-reduction story, plus (clearly labeled) internal pilots of ~30K msg/s, ~$1.2M savings, ~90% MIPS reduction; customers RBC / Alight / KeyBank.

★ UC-B — Db2 z/OS CDC for real-time analytics/AI flagship

Stream Db2 for z/OS active logs through IIDR or IBM Data Gate (Debezium-style, 96% zIIP) into Confluent, then Tableflow into watsonx.data for at-rest analytics and AI. This is the pattern that turns the mainframe system of record into a real-time feed without bolting query load onto Db2. Note the honest caveats: no fully-managed Db2-z/OS connector yet, and IIDR carries licensing cost.

UC-C — Confluent Platform on LinuxONE consolidation + hybrid

Mainframe sources → CDC → Confluent Platform running on LinuxONE (s390x, CFK on OpenShift) → Cluster Linking → Confluent Cloud → watsonx.data. You consolidate sprawling x86 streaming infrastructure onto IFLs (Signal65 TCO) while keeping a clean, offset-preserving bridge to the cloud for the AI/analytics side.
