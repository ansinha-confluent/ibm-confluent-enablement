The IBM + Confluent Technical Relationship

IBM completed its acquisition of Confluent on 17 March 2026. The two stacks are additive, not competing: Confluent supplies the real-time streaming substrate (Kafka + Flink); IBM watsonx supplies the open lakehouse, model serving, and agent orchestration that consume those streams. This page establishes the three-layer reference architecture, the open standards that bind the layers, and a decision rule for routing a workload to the right layer.

For an SE already fluent in Kafka and Flink, the mental model is straightforward: Confluent is the central nervous system for real-time data — IBM's own product documentation uses exactly that phrase. Events are produced once into Kafka and consumed many times: by stream processors (Flink/ksqlDB) for sub-minute reactions, by the lakehouse (watsonx.data via Tableflow → Iceberg) for at-rest analytics and AI training, and by agents (watsonx Orchestrate via MCP) for multi-step reasoning and action. Governance (watsonx.governance) and lineage span all three.

The three-layer reference architecture

Decision rule: which layer handles the workload?

The single most useful heuristic when an SE is whiteboarding with a customer:

Workload shapeLayer / productWhy
Sub-minute reaction to events; stateful stream joins/aggregationsConfluent — Flink / ksqlDBLowest latency; processes data in motion before it lands anywhere
Historical / at-rest analytics; AI training datasets; ad-hoc SQLwatsonx.data — Presto / Spark over IcebergColumnar, partitioned, queryable by many engines; cheap object storage
Multi-step workflow with human-in-the-loop; tool-calling agentswatsonx OrchestrateAgent supervision, delegation, approvals — not a stream-processing job
Semantic retrieval / RAG over enterprise documentswatsonx.data — MilvusVector similarity is a lakehouse-resident capability, fed by streams

Open standards are the integration contract

Every seam between the layers is an open standard, which is what makes the architecture composable rather than a set of proprietary bridges:

Apache Kafka

The event log and wire protocol. Every producer/consumer in the stack speaks it.

Apache Flink

Stream processing — both Confluent's managed Flink and IBM Event Processing's low-code UI compile to it.

Apache Iceberg + Parquet

Open table + file format. Tableflow materializes topics as Iceberg; watsonx.data reads them with zero copy.

Iceberg REST Catalog

The catalog protocol Tableflow exposes and Presto/Spark consume — no proprietary metastore lock-in.

Presto

Distributed SQL engine; watsonx.data ships a C++/Velox build for federated queries across Iceberg.

MCP (Model Context Protocol)

The tool-calling contract between Orchestrate agents and data services like ksqlDB-derived APIs.

Design consideration The acquisition does not collapse the products into one runtime. Confluent Cloud, Confluent Platform, and the watsonx family remain separate deployables with their own lifecycles. Integration happens at the standards boundary (Kafka protocol, Iceberg REST catalog, MCP), so an SE should architect each seam as a contract — version the schema, pin the catalog URI, and treat the agent's MCP tools as a stable API surface.

End-to-end pattern: stream → lakehouse → agent

A canonical agentic flow that touches all three layers, used later as the worked example on the Integration page: Kafka topics (inventory.transactions, inventory.availability) are aggregated by ksqlDB, exposed as an MCP tool get_sku_availability(sku, branch), and consumed by a watsonx Orchestrate supervisor agent that delegates to specialist agents and performs agentic RAG over enterprise documents held in watsonx.data Milvus.

IBM Confluent + IBM watsonx.data

Streaming into the Open Lakehouse

The integration point is Confluent Tableflow: it materializes a Kafka topic as an Apache Iceberg table, exposed through an Iceberg REST Catalog over object storage. watsonx.data then queries those tables in place — with its Presto (C++/Velox) and Spark engines — without copying data. This page covers the exact catalog wiring for both engines, the read-only constraint and write-back path, the Presto storage asymmetry, and the Milvus / lineage side-paths.

Tableflow turns "a topic" into "a table" continuously. Each topic becomes an Iceberg table whose schema is driven by the topic's Schema Registry subject; new records append as Iceberg data files, and schema changes propagate via Iceberg's schema-evolution. The result is a zero-copy federation surface: the same bytes that streamed through Kafka are queryable as columnar Parquet by any Iceberg-aware engine, with watsonx.data being the IBM-native one.

Wiring the Spark engine to the Tableflow catalog

watsonx.data's Spark engine registers Tableflow as a REST catalog. The full configuration block for Confluent-managed storage (remote-signing on, no customer credentials needed):

Spark — Tableflow REST catalog (Confluent-managed storage)
tableflow_config = {
    "spark.sql.catalog.tableflow": "org.apache.iceberg.spark.SparkCatalog",
    "spark.sql.catalog.tableflow.type": "rest",
    "spark.sql.catalog.tableflow.uri": "https://tableflow.{CLOUD_REGION}.aws.confluent.cloud/iceberg/catalog/organizations/{ORG_ID}/environments/{ENV_ID}",
    "spark.sql.catalog.tableflow.credential": "{APIKEY}:{SECRET}",
    "spark.sql.catalog.tableflow.io-impl": "org.apache.iceberg.aws.s3.S3FileIO",
    "spark.sql.catalog.tableflow.rest-metrics-reporting-enabled": "false",
    "spark.sql.catalog.tableflow.s3.remote-signing-enabled": "true",
    "spark.sql.catalog.tableflow.client.region": "{CLOUD_REGION}"
}

For customer-managed AWS S3 (you own the bucket), drop remote-signing and supply the S3 credentials directly by adding these keys:

Spark — additional keys for customer-managed S3
"spark.sql.catalog.tableflow.s3.access-key-id": "{S3_ACCESS_KEY}",
"spark.sql.catalog.tableflow.s3.secret-access-key": "{S3_SECRET_KEY}",
"spark.sql.catalog.tableflow.s3.region": "{CLOUD_REGION}"

Wiring the Presto engine to the Tableflow catalog

For Presto, you register a custom data source in watsonx.data's Infrastructure Manager with these connector properties. Note the OAuth2 auth type against the Iceberg REST endpoint:

Presto — custom data source connector properties
connector.name=iceberg
iceberg.catalog.type=REST
iceberg.rest.uri=https://tableflow.{CLOUD_REGION}.aws.confluent.cloud/iceberg/catalog/organizations/{ORG_ID}/environments/{ENV_ID}
iceberg.rest.auth.type=OAUTH2
iceberg.rest.auth.oauth2.credential={APIKEY}:{SECRET}
hive.s3.aws-access-key={S3_ACCESS_KEY}
hive.s3.aws-secret-key={S3_SECRET_KEY}

Querying is then ordinary Presto SQL — but the namespace is the Kafka cluster ID and must be double-quoted, and Tableflow exposes a "$timestamp" pseudo-column for event time:

Presto — querying Tableflow tables
SHOW SCHEMAS IN confluent_catalog;

SELECT * FROM confluent_catalog."{namespace}".{table_name} LIMIT 10;
-- {namespace} = Kafka cluster ID (e.g. "lkc-abc123"), must be double-quoted
Gotchas — Presto storage asymmetry & discovery
  • Presto requires customer-managed storage. Unlike Spark, the Presto connector does not support Confluent-managed storage / vended credentials — you must point it at your own S3 bucket with explicit keys. This is the single most common misconfiguration.
  • No auto-discovery. Tableflow tables do not appear in watsonx.data's Data Manager and are not enumerable via SHOW TABLES. You must know the exact topic name and Kafka cluster ID up front to query them.
  • Spark is more permissive: it supports both Confluent-managed and customer-managed storage. If a customer hasn't provisioned their own bucket yet, start with Spark.
Gotcha — read-only constraint & write-back path Tableflow tables are read-only from watsonx.data. You cannot INSERT/UPDATE against the Iceberg table to change source data — the only supported mutation path is to publish to the source Kafka topic, which Tableflow then materializes. Schema evolution is automatic, so additive schema changes flow through without manual catalog edits.

Side-paths: vector search and lineage

Milvus for RAG

watsonx.data embeds Milvus as its vector store. Streaming data landed in Iceberg can be embedded and indexed in Milvus for semantic retrieval; watsonx Orchestrate agents perform agentic RAG over that index. This is how the streaming substrate ultimately powers retrieval-augmented agents without a separate vector database to operate.

watsonx.data intelligence (Manta) lineage

watsonx.data intelligence harvests lineage from the Confluent Schema Registry through Tableflow into the Iceberg tables and downstream query engines, giving a column-level lineage graph that spans the streaming and at-rest halves of the architecture — the governance backbone introduced in the Overview.

Deployment topology note The object storage is the gravitational center. Co-locate watsonx.data's compute in the same cloud region as the Tableflow bucket (client.region / s3.region must match) to avoid cross-region egress on every scan. For Confluent-managed storage, Spark's remote-signing means watsonx.data never holds long-lived S3 keys — prefer it where your security posture disallows static credentials.

Real-world use cases

Three patterns that put the Tableflow → Iceberg → watsonx.data pipeline to work: zero-copy real-time BI, retrieval-augmented agents over a streaming-fed vector store, and a closed-loop ML feature/training pipeline. Each reuses the same materialized Iceberg substrate rather than copying data into a separate analytics silo.

UC 1 Real-time BI with zero-copy lakehouse federation

TableflowPrestoBI

Operational events (orders, clickstream, IoT) need to land in dashboards within seconds, not after an overnight batch. Producers write to Kafka with the Schema Registry enforcing the contract; Tableflow materializes the topic to an Iceberg table on object storage, and watsonx.data's Presto engine queries it directly. No ETL job copies the data into a warehouse — the Iceberg table is the warehouse table. [1][3]

Key technical details
  • Tableflow tables are read-only downstream and evolve schema automatically — additive changes flow through without manual catalog edits.
  • Presto requires customer-managed storage (no vended credentials); Spark works against both Confluent-managed and customer S3. Pick the engine to match your storage and credential posture.
  • Co-locate watsonx.data compute in the Tableflow bucket's region to avoid cross-region egress on every scan.

UC 2 Retrieval-augmented agents over a streaming-fed Milvus index

MilvusRAGOrchestrate

Knowledge that changes constantly (support tickets, product docs, policy updates) must reach an agent without a nightly re-index. Documents stream through Kafka, an embedding pipeline (watsonx.ai or Spark) vectorizes them, and the vectors land in watsonx.data's embedded Milvus store. A watsonx Orchestrate agent retrieves the nearest neighbors at query time for grounded, up-to-date answers — agentic RAG with no separate vector database to operate. [5]

Key technical details
  • Milvus ships inside watsonx.data, so the vector store sits next to the Iceberg tables it was derived from — one platform for both at-rest analytics and ANN search.
  • Because the index is fed by a stream, it stays current: a changed document re-embeds and updates the vectors rather than waiting for a scheduled rebuild.
  • Orchestrate retrieves over Milvus as a tool, making this RAG path composable with the other agent tools (e.g. the ksqlDB→MCP availability tool from the Overview).

UC 3 Closed-loop ML feature & training pipeline

FlinkSparkwatsonx.ai

A scoring model (fraud, churn, recommendations) needs both fresh features at inference time and a reproducible training set. Flink computes windowed features on the live stream; Tableflow lands the same events as Iceberg snapshots. watsonx.data's Spark engine does feature engineering over the time-travel snapshots, watsonx.ai trains, and the new model deploys back into the Flink scoring job — closing the loop. [2]

Key technical details
  • Iceberg time-travel snapshots give reproducible training sets — you can train against the exact data state as of a snapshot, eliminating train/serve skew from a moving table.
  • Flink owns the low-latency path (features + scoring); Spark owns the heavy batch path (feature engineering + backfills) — both read the same materialized substrate.
  • This is the training half of the Overview's fraud loop: the model deployed here is the scorer in that diagram, and the lakehouse sink there is the snapshot source here.

IBM Confluent + IBM Integration

Confluent in the IBM Integration Estate

IBM's integration portfolio — IBM MQ, webMethods/ACE, Event Automation, and StreamSets/DataStage — meets Confluent at well-defined connector and protocol boundaries. This page maps each integration product to its Confluent seam, walks the Event Automation layering and its Statement of Direction toward IBM Confluent Platform, covers the MQ source/sink trade-offs, and enumerates the GA connector set (Db2, Oracle XStream, SAP).

The organizing idea: Confluent Kafka is the durable event backbone, and the IBM integration products are either producers/consumers on that backbone (MQ, connectors) or processing and governance layers built over it (Event Automation). The acquisition's Statement of Direction consolidates IBM's own event-streaming components onto Confluent's engine over time.

Event Automation and the Statement of Direction

IBM Event Automation is three components: Event Streams (a managed Kafka distribution), Event Processing (a low-code authoring UI that compiles to Apache Flink), and Event Endpoint Management (an AsyncAPI-based catalog that governs and socializes event endpoints). Historically these shipped as add-ons to IBM Cloud Pak for Integration (CP4I).

Statement of Direction Post-acquisition, IBM's direction is to converge its event-streaming engine onto Confluent. Event Streams and Event Processing are slated for deprecation in future CP4I / webMethods releases, with future event-driven architecture delivered via IBM Confluent Platform — available both standalone and as an add-on to CP4I and webMethods. When advising a customer starting fresh today, architect against Confluent Platform / Confluent Cloud rather than net-new Event Streams deployments.

IBM MQ coexistence: source and sink trade-offs

The MQ connectors bridge queue semantics and topic semantics. The Source connector drains an MQ queue into a Kafka topic; the Sink connector writes a Kafka topic to an MQ queue. The design tension an SE must surface:

ConcernIBM MQKafkaImplication at the bridge
Consumption modelDestructive get (queue drains)Offset-based replay (log retained)Source connector turns a one-shot get into a replayable log — downstream can reprocess
OrderingPer-queue FIFOPer-partitionMap a queue to a single partition (or key carefully) to preserve order
DeliveryTransactional, once-onlyAt-least-once by defaultExpect possible duplicates after Kafka; dedupe downstream or use EOS where supported
Message size / formatJMS / native MQ, often EBCDIC from ZBytes + schemaConvert + register schema at ingest; see the IBM Z page for z/OS specifics

On the mainframe specifically, running the MQ connectors in bindings mode on zIIP avoids the CHINIT networking path and slashes the associated consumption cost — covered in depth on the IBM Z page.

Database and application connectors (GA set)

IBM Db2 Source (JDBC) — GA

Confluent Cloud managed connector for Db2; query/timestamp-based capture into Kafka. For Db2 for z/OS log-based CDC, see DataGate on the IBM Z page.

Oracle XStream CDC Source

Low-impact log-based CDC via Oracle's XStream API — the preferred high-fidelity Oracle capture path into Kafka.

SAP S/4HANA

Confluent + SAP partnership; a two-phase capture pattern for S/4HANA events into Kafka for downstream processing.

StreamSets / IBM Data Integration vs Confluent DSP

StreamSets and DataStage cover batch + pipeline ETL; Confluent's Stream Processing (Flink/ksqlDB) covers in-motion transformation. Use ETL for at-rest movement, DSP for sub-minute stream logic.

Worked example: event-driven agentic flow

Tying the integration layer to the agent layer — the canonical pattern from the Overview:

  1. Source systems publish to Kafka topics inventory.transactions and inventory.availability (via MQ/CDC connectors above).
  2. ksqlDB aggregates transactions into a live availability materialized view.
  3. The view is exposed as an MCP tool get_sku_availability(sku, branch).
  4. A watsonx Orchestrate supervisor agent (Store_Associate_Agent) delegates to specialists (SKU_Availability_Agent, Substitute_Finder_Agent) and performs agentic RAG over enterprise docs in watsonx.data Milvus.
Design consideration — connector placement Prefer fully-managed Confluent Cloud connectors where the source is reachable from the cloud; drop to self-managed Connect (or Connect on z/OS) only when the source is network-isolated (mainframe, on-prem MQ behind a firewall). Mixing managed and self-managed Connect clusters is fine, but keep one Schema Registry as the single source of truth for schemas across both.

Real-world use cases

Three integration patterns SEs hit repeatedly: a bi-directional MQ⇄Kafka bridge that keeps core systems in the loop, an Event Endpoint Management mesh that turns topics into governed, self-service products, and a StreamSets + DataStage ETL pattern that complements (not competes with) Confluent's in-stream processing.

UC 1 Bi-directional MQ ⇄ Kafka bridge

IBM MQSource/Sinkexactly-once

A core system speaks IBM MQ; the event-driven estate speaks Kafka. The MQ Source Connector drains a queue into a topic for Flink/microservice processing, and the MQ Sink Connector writes results back to an MQ queue the core system consumes — a closed bridge that modernizes around the mainframe without rewriting it. [3][4]

Key technical details
  • Both connectors are JMS-based and require the IBM MQ client JAR on the Connect worker (not bundled for licensing reasons).
  • Exactly-once: the Source supports at-least-once/exactly-once (v12.x); the Sink's exactly-once needs a dedicated offset queue set with DEFSOPT=EXCLUSIVE — it is not auto-created and must not be shared.
  • The Sink maps JMS headers (JMSCorrelationId, JMSReplyTo, JMSType) and supports a dead-letter queue plus multiple tasks for throughput.

UC 2 Governed event mesh with Event Endpoint Management

Event AutomationAsyncAPIself-service

Topics proliferate and nobody knows which are safe to consume. An admin registers a topic in the Event Manager, describes it with AsyncAPI + schema, and publishes it to the Event Catalog. Developers self-discover and request credentials; their clients connect through the Event Gateway, which enforces auth, quotas, redaction, and schema filtering at runtime — independent of the Kafka cluster, and equally over Confluent or Amazon MSK. [11]

Key technical details
  • EEM splits control plane (Event Manager — cataloging, policy) from data plane (Event Gateway — runtime enforcement), so policy changes don't touch the cluster.
  • Enforcement is broker-agnostic: the same governance applies whether the backing cluster is Confluent or Amazon MSK, which matters for multi-vendor estates.
  • Endpoints can be imported into API Connect, bridging the event mesh into the broader API management story.

UC 3 StreamSets + DataStage ETL alongside in-stream processing

StreamSetsDataStagewatsonx.data integration

Not everything belongs in a stream. StreamSets builds drift-aware smart pipelines into Kafka topics; DataStage runs batch/ELT (via a remote engine) into watsonx.data Iceberg; Flink does the sub-minute in-stream compute. A unified watsonx.data integration control plane gives shared observability, quality, and lineage across all three. [8][9]

Key technical details
  • Division of labor: ETL tools (StreamSets/DataStage) move data at rest; Confluent DSP (Flink/ksqlDB) transforms data in motion. They are complementary, not substitutes.
  • StreamSets' schema-drift handling keeps pipelines running when upstream shapes change — useful for feeding topics that the Schema Registry then governs.
  • The shared control plane is what makes the mix manageable: one place for lineage and quality across stream and batch, feeding the same Iceberg substrate the watsonx.data page queries.

IBM Confluent + IBM Z

Streaming the Mainframe

IBM Z carries the most valuable transactional data in the enterprise, historically trapped behind batch. Confluent unlocks it through three paths: log-based CDC (IBM Data Gate for Confluent, IIDR, zDIH), the certified Connect on z/OS runtime with IBM MQ connectors in bindings mode on zIIP, and Confluent Platform running on s390x (LinuxONE) with Cluster Linking to Confluent Cloud. This page covers the topology, the zIIP economics, and the exact z/OS setup details (versions, EBCDIC conversion, JCL, bindings).

Path 1 — Log-based CDC from Db2 for z/OS

IBM Data Gate for Confluent (GA 2026-06-12) is the native, Confluent-first path: it captures changes from Db2 for z/OS using log-based CDC, delivers them through native Kafka Connect, and supports both initial snapshot and continuous streaming. The headline efficiency number for SEs:

zIIP economics IBM Data Gate for Confluent is up to 96% zIIP-eligible. Because eligible Java workloads on the zIIP specialty engine are not counted toward general-purpose CP (and thus not chargeable under MSU-based licensing), pushing CDC onto zIIP is what makes continuous mainframe streaming economically viable versus batch extracts. IIDR and zDIH are the alternative CDC routes; CICS→MQ→Kafka is the pattern when the system of record only emits through MQ.

EBCDIC: the property and log files must be IBM-1047

z/OS is an EBCDIC platform. The properties files you author (often in ASCII/ISO8859-1) must be transcoded to IBM-1047 before Connect can read them:

USS — transcode a properties file ASCII → EBCDIC (IBM-1047)
iconv -f ISO8859-1 -t IBM-1047 ./connect-standalone.properties > connect-standalone2.properties

The Log4j2 console appender must also be told to emit IBM-1047, or your logs are mojibake:

connect-log4j2.yaml — Console appender PatternLayout
charset: "IBM-1047"

JCL: redirect Connect stdout to a USS text file

When launching Connect from JCL, redirect stdout to a USS path with FILEDATA=TEXT so the bytes are treated as text (and newline-converted) rather than binary:

JCL — STDOUT redirect to USS
//STDOUT DD PATH='/u/kafconn/logs/kafconnect.out',
//          PATHOPTS=(OWRONLY,OCREAT,OAPPEND),
//          PATHMODE=(SIRWXU),
//          FILEDATA=TEXT

Bindings mode: point LIBPATH at the MQ client library

Bindings mode talks to the local queue manager through native libraries instead of TCP/CHINIT. Export the library path, then launch standalone Connect with the connector properties:

USS — bindings mode launch
export LIBPATH=$LIBPATH:<path_to_client_library>
./bin/connect-standalone.sh ./config/connect-standalone.properties ./ibmmqconnector.properties
Gotchas — z/OS Connect
  • Bindings vs client mode: bindings is the default and preferred — it avoids the CHINIT address space and its networking cost, and is more secure. Use client mode only when Connect and the queue manager are on different LPARs/hosts.
  • Every text artifact is EBCDIC: properties, log config, and JCL-written files. A single un-transcoded properties file produces silent parse failures.
  • zIIP is a prerequisite, not just an optimization for cost-effective operation — confirm zIIP capacity before sizing.

Path 3 — Confluent Platform on s390x + hybrid Cluster Linking

Confluent Platform is supported on s390x (Linux on IBM Z / LinuxONE, including zCX), deployable via Confluent for Kubernetes (CFK). This keeps brokers and processing physically next to the mainframe for data residency and lowest-latency capture. To extend to cloud-native consumers (watsonx analytics, AI, agents), Cluster Linking mirrors topics from the on-prem Confluent Platform cluster to Confluent Cloud over a single byte-for-byte replication link — no Connect cluster, offsets preserved.

Topology decision Use on-prem CP on LinuxONE when regulation or latency demands the streaming substrate stay next to Z; use Cluster Linking (not MirrorMaker, not a Connect bridge) to project a curated subset of topics to the cloud. Keep the sensitive topics local and link only what downstream cloud workloads need — Cluster Linking is topic-granular, so it doubles as a data-egress control point.

Real-world use cases

Three mainframe patterns, from newest to most battle-tested: log-based Db2 z/OS CDC into real-time fraud scoring via Data Gate, a forward-cache offload that takes read load off the mainframe, and the decoupling pattern that large banks have run in production for years.

UC 1 Real-time Db2 z/OS fraud detection via Data Gate

Data Gatelog-based CDCzIIP

Fraud decisions must happen on live mainframe transactions, not yesterday's batch extract. IBM Data Gate for Confluent captures changes from Db2 for z/OS with log-based CDC (snapshot + continuous), streams them through native Kafka Connect into Confluent Platform, where Flink/ksqlDB scores each event and emits an alert stream — while every scored event also sinks to the lakehouse for retraining. [3][4]

Key technical details
  • Log-based capture means minimal impact on core transactional workloads — no triggers, no polling against the live tables.
  • Up to 96% of Data Gate processing is zIIP-eligible, which keeps general-purpose CP consumption (and software bill) low at high throughput.
  • Snapshot + continuous streaming lets you onboard the existing dataset once and then stay continuously synchronized — the alert path and the lakehouse retraining path share one stream.

UC 2 Mainframe offload with a real-time forward cache

forward cachecost offloadMQ / CDC

Customer-facing apps hammering the mainframe for reads drive up MIPS cost and latency. Streaming back-end changes (via MQ Source or CDC) through Confluent Platform into a near-real-time forward cache lets consumer apps read the materialized view instead of the mainframe — Alight Solutions used exactly this pattern to cut mainframe cost and accelerate delivery. [2]

Key technical details
  • The cache is a read offload, not a system of record — the mainframe stays authoritative; only the read traffic moves.
  • Because many consumers read the same cache, you decouple consumer scaling from mainframe capacity — adding apps no longer adds mainframe load.
  • The same Confluent pipelines that feed the cache can fan out to analytics and AI, so the offload investment doubles as the ingestion layer for the rest of the architecture.

UC 3 System decoupling for digital transformation (RBC, KeyBank)

decouplingevent-drivenreuse

Large banks use Confluent Platform as the decoupling layer between the mainframe and new digital products. RBC wrote a microservice that serves reads off Confluent (reducing mainframe reads and OpEx) and dropped anomaly detection from weeks to seconds; KeyBank built a streaming center of excellence to decouple systems and reuse data, cutting time-to-market roughly in half.
